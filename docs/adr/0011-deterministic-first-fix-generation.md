# ADR-0011: Deterministic-first fix generation

**Status:** Accepted
**Date:** 2026-07-19

## Context

ADR-0001 drew a hard line on the read side: a parser finds the issue, never an LLM. Sprint 2 opened the write side, and the same temptation returned in a more dangerous form. It is one thing to hallucinate a finding; it is another to hallucinate a change and open a pull request that edits a client's repository. "Ask the model to rewrite this file so the canonical is correct" is fast to prototype and impossible to trust: it can reformat the whole file, drop unrelated content, invent a framework convention, or quietly change something nobody asked it to touch. A reviewer then has to diff the model against the world instead of reading a small, obvious change.

Meanwhile, almost every fix we ship is as mechanical as the detection was. Rewriting a canonical's origin, flipping a `Disallow: /` to `Allow: /`, removing a `noindex` token from a meta tag, inserting a tag before `</head>`: these are structured transforms of text a parser can locate, not acts of reasoning.

## Decision

**A fixer is a pure function of the finding and the repository, and it transforms structure it located rather than guessing.** ADR-0001 extended to the write side.

Concretely, in `packages/fixers`:

- A `Fixer` has two methods: `canFix(finding)` and `generate(ctx)`, where `ctx` carries the finding, the detected framework, and a `read(path)` closure. It returns whole-file `FileChange`s or `null`. It performs no network of its own and holds no database handle.
- Fixers edit through located, bounded transforms, never blind string replacement. The canonical fixer rewrites an origin only at a URL boundary, so `https://site.com` never corrupts `https://site.com.evil.test`. The robots fixer walks the file's groups and changes the one line that blocks an agent, keeping every other byte. The head injector finds a real `</head>` and inserts before it, matching the existing indentation.
- **When a fixer cannot locate what it would change, it returns `null`.** It never invents a location. A canonical whose origin is not in any head file, a robots.txt generated by code rather than a file, a `noindex` delivered by a response header: each yields `null`, and the worker reports honestly that no fix could be generated rather than opening a pull request that changes nothing or the wrong thing.
- The one place an LLM writes to a repository (`generateContentFix`, the meta-description writer) is held to the same shape. A deterministic rule (TECH-021) finds the issue; the LLM makes exactly one `smart` call via `llm.object` with a Zod schema; the output is validated before it can become a diff; and a deterministic head injection places it. The model writes text, a parser writes files. If the chain is unavailable the finding stays open (ADR-0005).

Every fix is therefore a small, reviewable, whole-file diff a human can read and merge, not a model's opinion about a file.

## Consequences

### Good

- Fixers are pure functions, unit tested against an in-memory repository with no network and no model. The canonical, robots, and noindex fixers each ship with a triggering and a clean fixture.
- A fix cannot silently touch what it did not intend to. The transform is bounded to the located structure.
- A fixer that is unsure does nothing, which is the safe failure for a tool that edits other people's code. Honest `null` beats a confident wrong diff.
- The LLM's blast radius is a single validated string dropped into a known place, not a rewritten file.

### Bad

- More engineering than prompting a model to "fix the file", and a fixer per rule rather than one universal fixer.
- Fixers that need per-route context (a distinct title for each of several pages) do not fit a single-file transform and are deliberately out of scope until the write path can map a URL to its source file. We say so rather than shipping a fixer that edits the wrong file.

### Neutral

- Forces the same question ADR-0001 forced, now for writes: where does the model actually add value? The answer is narrow (subjective text), and naming it narrowly is a feature.

## Alternatives considered

### LLM-as-fixer: send the file and the finding, apply the model's rewrite

Rejected, for the write-side version of every reason in ADR-0001, plus a sharper one: the output is a change to a client's repository, so a hallucination is not a bad row in a dashboard, it is a bad commit in a pull request against their `main`. Non-reproducible, expensive per file, and it reformats and drops content in ways a reviewer cannot easily catch.

### Blind string replacement for the deterministic fixers

Rejected. Replacing `https://site.com` everywhere, or `Disallow: /` everywhere, is deterministic but not bounded: it corrupts look-alike hostnames and rewrites lines in groups it should not touch. The transforms are located (a URL boundary, a specific agent's group) precisely so the diff is exactly the change and nothing more.

### Let the LLM fixer parse its own free-text output

Rejected by ADR-0005 already, and reaffirmed here: the content fixer uses `generateObject` with a schema, and the code parses the schema-validated object, never free text.
